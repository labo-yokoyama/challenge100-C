# WPF MVVM アプリケーション 技術仕様・共通設計ガイドライン

本ドキュメントは、WPF (Windows Presentation Foundation) と MVVM パターンを用いたアプリケーション開発における、標準的なアーキテクチャ、設計原則、技術的制約、および実装上の注意点を定義します。

## 1. アーキテクチャ概要

本プロジェクトは **WPF (Windows Presentation Foundation)** と **MVVM (Model-View-ViewModel)** パターンを採用しています。
MVVM インフラストラクチャとして **Livet** フレームワークを使用し、UIスレッド管理やメッセージング（ダイアログ表示）を効率化しています。

### コンポーネント構成
- **View (XAML)**: UIレイアウトとデータバインディング。UIの定義。ロジック（コードビハインド）を排除し、データバインディングのみで ViewModel と通信します。
- **ViewModel**: アプリケーションの状態管理、コマンド、ビジネスロジックの橋渡し。UIの状態と振る舞いを保持します。`INotifyPropertyChanged` を介して View を更新し、`ICommand` を介してユーザー操作を受け取ります。
- **Model**: ファイルI/O、シリアライズ処理。純粋なビジネスロジック、計算処理（単位変換など）、またはデータの永続化（JSON保存など）を担当します。UIの概念を持ちません。

## 2. 技術スタック

- **Framework**: .NET Framework4.5.2 (WPF)
- **MVVM Library**: Livet
- **言語**: C# 5.0以前

## 3. 実装の主要なポイント

### 3.1 初期化処理（Attached Behavior）
- **ContentRendered**: ウィンドウが表示された直後に初期化（ディレクトリ作成、ファイル読み込み）を行うため、カスタム添付プロパティ `WindowBehaviors.ContentRenderedCommand` を使用しています。これにより、コンストラクタではなくUI準備完了タイミングでの処理実行を保証しています。

### 3.2 例外処理
- 各メソッド内での個別の try-catch だけでなく、`App.xaml.cs` において以下のグローバル例外ハンドラを実装することを推奨します。
    - `DispatcherUnhandledException`: UIスレッドでのクラッシュ防止。
    - `AppDomain.CurrentDomain.UnhandledException`: 非UIスレッド等を含む全般的な例外の捕捉。

### 3.3 UIメッセージング
- ViewModel から `MessageBox` を直接呼び出すのは避けます。
- **方法1 (Livet等)**: `Messenger` パターンを使用し、ViewModel がメッセージを発信し、View がそれを受けてダイアログを表示します。
- **方法2 (サービス注入)**: ダイアログ表示用のインターフェースを Model またはサービスとして ViewModel に注入します。

### 3.4 バインディングの最適化
- **UpdateSourceTrigger**: テキストボックスの入力と同時に処理を行いたい場合（リアルタイム検索やリアルタイム変換など）は `PropertyChanged` を指定します。
- **Mode**: 読み取り専用のプロパティには `OneWay` を、入力項目には `TwoWay` を適切に設定し、予期せぬデータ更新を防ぎます。

## 4. コーディング規約
- フィールド先頭にアンダースコア（_）を付けない。
- すべてのクラス、プロパティ、メソッドに <summary> コメントを付ける。
  - メソッドは、引数や戻り値が存在する場合のみ <param> や <returns> を記述する。存在しない場合は記述自体を省略する。
  - プロパティの summary は「メモのタイトル」のように名詞句で記述する（「～を取得または設定します」は不要）。
- 文字化けを防ぐため、ファイル入出力時は UTF-8 を明示的に指定する。
- C# 5.0 で使用できない文法は絶対に使用しない。
- 未使用の `using` は削除すること。
