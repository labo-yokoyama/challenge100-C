# 07_JsonNotepad 技術仕様・実装ガイドライン

このドキュメントは、JSONファイル形式でデータを保存するデスクトップ・メモアプリ（JsonNotepad）の設計構造、技術的制約、および実装上の注意点をまとめたものです。

## 1. アーキテクチャ概要

本プロジェクトは **WPF (Windows Presentation Foundation)** と **MVVM (Model-View-ViewModel)** パターンを採用しています。
MVVM インフラストラクチャとして **Livet** フレームワークを使用し、UIスレッド管理やメッセージング（ダイアログ表示）を効率化しています。

### コンポーネント構成
- **View**: `MainWindow.xaml` - UIレイアウトとデータバインディング。
- **ViewModel**: `MainViewModel.cs` - アプリケーションの状態管理、コマンド、ビジネスロジックの橋渡し。
- **Model**: `JsonNotepadModel.cs` - ファイルI/O、シリアライズ処理。
- **Data**: `NoteItem.cs` - メモのデータ構造（POCO）。

## 2. 技術スタック

- **Framework**: .NET (WPF)
- **MVVM Library**: Livet
- **JSON Library**: Newtonsoft.Json (Json.NET)
- **Storage**: ローカルファイルシステム（`.json` ファイル）

## 3. 実装の主要なポイント

### 3.1 データの保存と同期
- **ファイルベースの管理**: 各メモを個別の `.json` ファイルとして保存します。
- **ファイル名とタイトルの同期**: メモの「タイトル」をそのままファイル名として使用します。
    - タイトルが変更された場合、古いファイルを削除し、新しいタイトルで保存し直すことで「リネーム」を実現しています。
- **シリアライズ**: `Newtonsoft.Json` を使用し、UTF-8エンコーディングで保存します。

### 3.2 検索とフィルタリング
- **メモリ内フィルタリング**: 保存されているすべてのメモを `allNotes` (ObservableCollection) に保持し、検索文字列 (`SearchText`) の変更に合わせて `FilteredNotes` を動的に更新します。
- **リアルタイム検索**: `SearchText` プロパティの `set` アクセサ内でフィルタ処理を実行することで、入力と同時に一覧が更新される仕組みです。

### 3.3 初期化処理（Attached Behavior）
- **ContentRendered**: ウィンドウが表示された直後に初期化（ディレクトリ作成、ファイル読み込み）を行うため、カスタム添付プロパティ `WindowBehaviors.ContentRenderedCommand` を使用しています。これにより、コンストラクタではなくUI準備完了タイミングでの処理実行を保証しています。

### 3.4 例外処理
- `App.xaml.cs` にて、UIスレッド (`DispatcherUnhandledException`) および非UIスレッド (`AppDomain.CurrentDomain.UnhandledException`) の両方で未処理例外をキャッチし、クラッシュ前にユーザーへエラー内容を通知する仕組みを備えています。

## 4. 制約事項と注意点

### 4.1 ファイル名の制約
- タイトルをファイル名として利用するため、OSが禁止している文字（`\`, `/`, `:`, `*`, `?`, `"`, `<`, `>`, `|`）はバリデーションで弾く必要があります。
- `Path.GetInvalidFileNameChars()` を用いたチェックを推奨します。

### 4.2 ディレクトリ管理
- 実行ファイルの場所（`AppDomain.CurrentDomain.BaseDirectory`）を基準に `notes` フォルダを作成します。
- アプリ起動時にディレクトリの存在確認と作成 (`Directory.CreateDirectory`) が必要です。

### 4.3 コマンドの活性・非活性管理
- 削除コマンドなど、選択状態に依存する処理は `CanExecute` (または Livet の `RaiseCanExecuteChanged`) を適切に呼び出し、UIのボタン状態と同期させる必要があります。

### 4.4 ダイアログの表示
- ViewModel から直接 `MessageBox` を呼ぶのではなく、Livet の `Messenger` 経由でメッセージを飛ばし、View 側でダイアログを表示させることで、ViewModel のテスト容易性を維持します。

## 5. 拡張へのヒント

同系統のアプリを作成する際の拡張アイデア：
1. **自動保存機能**: タイピングが一定時間停止した際に自動的に `SaveNote` を実行する。
2. **カテゴリ・タグ機能**: JSON 構造に `Tags` プロパティを追加し、フィルタリング条件を拡張する。
3. **バックアップ**: 保存時に `notes/backup` フォルダに過去バージョンを保持する。
4. **エディタの強化**: Markdown プレビュー機能の追加など。

- プロパティ値の変更をViewに通知するための共通基底クラスです。
- `OnPropertyChanged` または `RaisePropertyChanged` メソッドを提供します。

### 2.2 RelayCommand / ViewModelCommand (`ICommand`)
- ボタンクリックなどのイベントを ViewModel のメソッドにバインドするための汎用クラスです。
- `CanExecute` により、ボタンの有効/無効状態を動的に制御します（例：入力値が不正な場合に計算ボタンを無効化する）。

## 3. 実装ガイドライン

### 3.1 ユーザー入力の検証 (Validation)
- コマンド実行前に ViewModel 側で入力をチェックします。
- **データ管理系 (JsonNotepad)**: ファイル名として不適切な文字のチェック。
- **ロジック計算系 (UnitConverter)**: 数値変換が可能か、正の数であるか等のチェック。

### 3.2 UIイベントのコマンド化 (Attached Behavior)
- `Window.Loaded` や `ContentRendered` など、XAMLの標準バインディングでは対応できないイベントを ViewModel のコマンドに接続する場合は、添付プロパティ（Attached Behavior）を使用します。
- これにより、初期化処理を ViewModel のライフサイクルに統合できます。

### 3.3 非同期処理とスレッド管理
- I/O（ファイル入出力）や重い計算を行う際は、UIスレッドをブロックしないよう `async/await` の利用を検討します。
- 必要に応じて `Dispatcher` を介して結果をUIに書き戻します。

### 3.4 例外処理の共通化
- 各メソッド内での個別の try-catch だけでなく、`App.xaml.cs` において以下のグローバル例外ハンドラを実装することを推奨します。
    - `DispatcherUnhandledException`: UIスレッドでのクラッシュ防止。
    - `AppDomain.CurrentDomain.UnhandledException`: 非UIスレッド等を含む全般的な例外の捕捉。

## 4. プロジェクト種別ごとの Model 実装指針

### 4.1 状態保持・永続化型 (例: Notepad, Todoリスト)
- **責務**: ファイルの読み書き、シリアライズ (JSON/XML)、ディレクトリの初期化。
- **ポイント**: ファイルシステムとの同期、アトミックな書き込み（保存失敗時のデータ保護）。

### 4.2 ロジック・計算型 (例: UnitConverter, 計算機)
- **責務**: 数値変換、ビジネスルールの適用、外部APIとの通信。
- **ポイント**: UIに依存しない純粋な計算ロジック。入力値に対する副作用のない戻り値の提供。

## 5. UIメッセージング

- ViewModel から `MessageBox` を直接呼び出すのは避けます。
- **方法1 (Livet等)**: `Messenger` パターンを使用し、ViewModel がメッセージを発信し、View がそれを受けてダイアログを表示します。
- **方法2 (サービス注入)**: ダイアログ表示用のインターフェースを Model またはサービスとして ViewModel に注入します。

## 6. バインディングの最適化

- **UpdateSourceTrigger**: テキストボックスの入力と同時に処理を行いたい場合（リアルタイム検索やリアルタイム変換など）は `PropertyChanged` を指定します。
- **Mode**: 読み取り専用のプロパティには `OneWay` を、入力項目には `TwoWay` を適切に設定し、予期せぬデータ更新を防ぎます。
